# 강화 리뷰 루프 (Reinforce)

7개 전문 에이전트가 코드를 리뷰하고, Critical/Major 이슈를 자동 수정한 뒤, 다시 리뷰하는 과정을 **이슈가 해소될 때까지 반복**하는 스킬입니다.

**GitHub에 코멘트를 남기지 않습니다.** 모든 과정은 대화 내에서 이루어집니다.

**모든 출력은 한국어로 작성한다.**

## 사용 방법

```text
/roto-band-reinforce [PR_NUMBER] [--max-rounds N]
```

| 인자 | 설명 |
|------|------|
| `PR_NUMBER` | GitHub PR 번호 (선택). 생략 시 현재 브랜치의 PR을 자동 탐지 |
| `--max-rounds N` | 최대 반복 횟수 (선택, 기본값: 3, 최대: 5) |

### 예시

```text
/roto-band-reinforce              # 현재 브랜치 PR, 최대 3회
/roto-band-reinforce 64           # PR #64, 최대 3회
/roto-band-reinforce --max-rounds 5  # 최대 5회 반복
```

## 워크플로우

### Step 1: 인자 파싱 및 PR 준비

1. `--max-rounds N` 파싱. 없으면 기본값 3. 최대 5 초과 시 5로 클램프.
2. PR 번호 결정: 인자 숫자 또는 `gh pr view --json number --jq .number` 자동 탐지.
3. PR 메타데이터 수집:
   ```bash
   gh pr view <PR_NUMBER> --json number,title,additions,deletions,changedFiles,files,baseRefName,headRefName,headRefOid
   ```
4. 리포트 파일 경로 결정: `code-reviews/reinforce-PR-<PR_NUMBER>.md`
5. 사용자에게 시작 알림:
   ```
   🔄 PR #<NUMBER> 강화 리뷰를 시작합니다. (최대 N 라운드)
   ```

### Step 2: 초기 리뷰 (Round 0)

아래 **리뷰 절차**를 실행하여 첫 번째 리뷰 결과를 얻는다.

결과를 리포트 파일에 저장하고, 사용자에게 요약을 출력한다:

```
📋 Round 0 (초기 리뷰): 🔴 N / 🟠 N / 🟡 N / 🔵 N
```

### Step 3: 검증 리뷰 (3회)

초기 리뷰 결과에 대해 **3회 추가 검증 리뷰**를 수행한다.

각 검증 리뷰에서는:
1. 초기 리뷰에서 보고된 이슈 목록을 에이전트에게 함께 전달
2. 에이전트는 해당 이슈가 **실제로 타당한지**, **오탐(false positive)인지** 판단
3. 새로운 이슈가 발견되면 추가

검증 리뷰 프롬프트에 아래 지시를 추가한다:

```
아래는 이전 리뷰에서 보고된 이슈 목록입니다. 각 이슈에 대해:
1. 실제 문제가 맞는지 코드를 직접 확인하세요
2. 오탐(false positive)이면 해당 이슈를 보고하지 마세요
3. 새로운 이슈가 있으면 추가로 보고하세요

<이전 이슈 목록>
```

3회 검증 후 이슈 통합:
- 4회 중 **2회 이상** 보고된 이슈만 확정 이슈로 채택
- 1회만 보고된 이슈는 `의심` 태그를 달고 Minor로 강등
- 각 이슈에 `confidence` 필드 추가 (보고 횟수 / 4)

통합 결과를 리포트에 기록하고 사용자에게 출력:

```
🔍 검증 완료: 확정 이슈 N건 (🔴 N / 🟠 N), 제거된 오탐 N건
```

Critical 또는 Major가 **0건**이면 → Step 6(최종 요약)으로 건너뛴다.

### Step 4: 자동 수정 라운드

확정된 Critical/Major 이슈를 자동으로 수정한다.

**수정 대상**: Critical + Major만. Minor/Misc는 수정하지 않는다.

**수정 절차** (이슈 하나당):
1. Read 도구로 해당 파일의 현재 코드 확인
2. 리뷰 권장안을 기반으로 Edit/Write 도구로 수정
3. 수정이 다른 이슈에 영향을 줄 수 있으므로 후속 이슈의 코드를 다시 확인

**수정 원칙**:
- 수정 범위는 리뷰 지적 사항으로 한정. 주변 코드를 불필요하게 리팩터링하지 않는다.
- import 추가/제거가 필요하면 함께 처리
- 수정 중 에러 발생 시 해당 이슈 스킵 + 사유 기록

수정 완료 후:
- TypeScript 타입 체크: `npx tsc --noEmit`
- 테스트: 프로젝트의 테스트 커맨드 실행
- 에러 발생 시 수정 시도

리포트에 수정 내역 기록:

```
🔧 Round 1 수정: ✅ N건 수정 / ⏭️ N건 스킵
```

### Step 5: 재리뷰 → 수정 반복

수정 후 다시 **리뷰 절차**를 실행한다. 이때 diff를 새로 수집한다:

```bash
gh pr diff <PR_NUMBER>
```

> **주의**: PR에 아직 push하지 않은 로컬 변경이 있으므로, 로컬 diff를 사용한다:
> ```bash
> git diff <BASE_BRANCH>...HEAD
> ```

리뷰 결과에서:
- Critical 또는 Major가 **있으면** → Step 4로 돌아가 수정 반복
- Critical/Major가 **0건**이면 → Step 6으로 진행
- `max-rounds`에 도달하면 → Step 6으로 진행 (잔여 이슈 보고)

각 라운드마다 사용자에게 진행 상황 출력:

```
🔄 Round N 리뷰: 🔴 N / 🟠 N / 🟡 N / 🔵 N → 수정 진행
```
또는
```
✅ Round N 리뷰: Critical/Major 0건 — 강화 완료!
```

### Step 6: 최종 요약 및 리포트 저장

#### 6-1: 리포트 최종 업데이트

리포트 파일(`code-reviews/reinforce-PR-<PR_NUMBER>.md`)에 전체 과정을 기록한다.

리포트 템플릿:

```markdown
# PR-<NUMBER> 강화 리뷰 리포트

## 메타데이터

| 항목 | 값 |
|------|-----|
| PR | #<NUMBER> |
| 리뷰 일시 | <YYYY-MM-DD HH:MM> |
| 리뷰어 | AI Multi-Agent (`/roto-band-reinforce`) |
| 총 라운드 | N |
| 최종 상태 | ✅ 해소 / ⚠️ 잔여 이슈 있음 |

## 라운드별 진행

### Round 0: 초기 리뷰
| Severity | 개수 |
|----------|------|
| 🔴 Critical | N |
| 🟠 Major | N |
| 🟡 Minor | N |
| 🔵 Misc | N |

#### 보고된 이슈
| # | Severity | 파일 | 라인 | 에이전트 | 제목 | confidence |
|---|----------|------|------|----------|------|------------|
| 1 | 🔴 | file.ts | 42 | Senior FE | 이슈 제목 | 4/4 |

<각 이슈 상세>

### 검증 결과
- 확정 이슈: N건
- 제거된 오탐: N건

### Round 1: 수정 + 재리뷰
#### 수정 내역
| # | 이슈 | 상태 | 수정 내용 |
|---|------|------|----------|
| 1 | Open Redirect | ✅ 수정 | URL 검증 추가 |
| 2 | XSS 취약점 | ✅ 수정 | sanitize 적용 |

#### 재리뷰 결과
| Severity | 개수 |
|----------|------|
| 🔴 Critical | 0 |
| 🟠 Major | 0 |
| 🟡 Minor | N |
| 🔵 Misc | N |

(라운드별 반복)

## 최종 결과

### 해소된 이슈
- ✅ [에이전트명] 이슈 제목 — `파일:라인` (Round N에서 수정)

### 잔여 이슈 (Minor/Misc)
- 🟡 [에이전트명] 이슈 제목 — `파일:라인`

### 잔여 이슈 (미해소 Critical/Major) — max-rounds 도달 시만
- 🔴 [에이전트명] 이슈 제목 — `파일:라인`

### 수정된 파일
- `path/to/file1.ts`
- `path/to/file2.tsx`

### 검증 결과
- TypeScript: ✅ 통과
- 테스트: ✅ N개 통과

---
🤖 Generated by `/roto-band-reinforce` | iterative multi-agent review
```

## 리뷰 절차 (공통)

각 라운드에서 실행하는 리뷰 절차는 다음과 같다. `/roto-band-code-review`의 Step 3~5를 silent 모드로 실행하는 것과 동일하다.

1. **Diff 수집**: `git diff <BASE_BRANCH>...HEAD` (로컬 변경 포함)
2. **파일 필터링**: 자동 제외 패턴 적용 (`*.lock`, `*.min.*`, `*.map`, `*.generated.*`, `dist/`, `build/`, `.next/`, `*.snap`, `*.svg`)
3. **7개 서브 에이전트 병렬 실행**: Agent 도구로 동시 spawn

   | 에이전트 | 파일 | model |
   |----------|------|-------|
   | UI Designer | `agents/ui-designer.md` | sonnet |
   | Senior FE | `agents/senior-fe-engineer.md` | sonnet |
   | Junior FE | `agents/junior-fe-engineer.md` | sonnet |
   | BE Engineer | `agents/backend-engineer.md` | sonnet |
   | SEO Specialist | `agents/seo-specialist.md` | sonnet |
   | i18n Reviewer | `agents/i18n-reviewer.md` | sonnet |
   | Claude Config | `agents/claude-config-reviewer.md` | sonnet |

4. **결과 통합**: 파싱 → 중복 통합 (±3라인) → severity 정렬

## 도구 사용 규칙

Bash 대신 Claude 전용 도구를 우선 사용한다:

| 작업 | 사용할 도구 | Bash 사용 금지 |
|------|-----------|---------------|
| 파일 읽기 | **Read** | `cat`, `head`, `tail` |
| 파일 수정 | **Edit** | `sed`, `awk` |
| 파일 생성 | **Write** (디렉토리 자동 생성) | `mkdir -p` + `echo >` |
| 파일 검색 | **Glob** | `find`, `ls` |
| 코드 내용 검색 | **Grep** | `grep`, `rg` |
| 리포트 업데이트 | **Edit** | `sed` |

Bash는 아래 경우에만 사용한다:
- `gh` CLI 명령 (PR 조회)
- `git diff` (로컬 diff 수집)
- `npx tsc --noEmit` (타입 체크)
- 프로젝트 테스트 명령

## 오류 처리

| 상황 | 처리 |
|------|------|
| GitHub 인증 실패 | `[ERROR] GitHub authentication required. Run: gh auth login` |
| PR 미존재 | `[ERROR] PR #<NUMBER> not found.` |
| 서브 에이전트 전원 실패 | `[ERROR] All agents failed. Please retry.` |
| 일부 에이전트 실패 | 실패 에이전트 이름 로그, 나머지 결과로 진행 |
| 초기 리뷰 이슈 0건 | `✅ 리뷰 결과 이슈가 없습니다. 코드가 양호합니다!` 출력 후 종료 |
| 수정 중 에러 | 해당 이슈 스킵 + 사유 기록, 나머지 계속 |
| 타입 체크/테스트 실패 | 수정 시도, 실패 시 사유 기록 후 계속 |
| max-rounds 도달 | 잔여 이슈 보고 후 종료 |
| 동일 이슈 반복 발생 | 2라운드 연속 같은 file+line+title 이슈 시 해당 이슈 스킵 (무한 루프 방지) |
