# 코드 리뷰 반영 스킬

PR의 인라인 리뷰 코멘트를 수집 → 타당성 검토 → 코드 수정 → 수정 완료 댓글 → PR 요약 코멘트까지 자동으로 처리합니다.

**모든 출력은 한국어로 작성한다.** PR 코멘트, 인라인 답글, 요약 코멘트, 대화 창 출력, 작업 계획 테이블 등 사용자에게 보이는 모든 텍스트는 한국어를 사용할 것. 코드 주석이나 변수명 등 코드 자체는 기존 코드베이스의 언어 규칙을 따른다.

## 사용 방법

```text
/roto-band-review-apply [PR_NUMBER]
```

| 인자 | 설명 |
|------|------|
| `PR_NUMBER` | GitHub PR 번호 (선택). 생략 시 현재 브랜치의 PR을 자동 탐지 |

---

## 워크플로우

### Step 1: PR 번호 결정 및 메타데이터 수집

1. 인자에서 숫자가 있으면 `PR_NUMBER`로 사용한다.
2. **PR 번호가 생략된 경우**, 현재 브랜치에서 자동 탐지:
   ```bash
   gh pr view --json number --jq .number
   ```
   - 실패 시 에러: `[ERROR] 현재 브랜치에 연결된 PR이 없습니다. PR 번호를 직접 지정해주세요.`
3. owner/repo 확인:
   ```bash
   gh repo view --json nameWithOwner --jq .nameWithOwner
   ```
4. PR 메타데이터 수집:
   ```bash
   gh pr view <PR_NUMBER> --json number,title,headRefName,headRefOid,baseRefName
   ```
5. 현재 브랜치가 PR의 head 브랜치와 다르면, 해당 브랜치로 전환할지 사용자에게 확인한다.

### Step 2: 인라인 리뷰 코멘트 수집

PR에 남겨진 모든 인라인 리뷰 코멘트를 가져온다:

```bash
gh api repos/{owner}/{repo}/pulls/<PR_NUMBER>/comments --paginate
```

#### 코멘트 파싱

각 코멘트에서 다음 정보를 추출한다:

| 필드 | JSON 경로 | 용도 |
|------|-----------|------|
| `id` | `.id` | 댓글 회신 시 사용 |
| `path` | `.path` | 수정 대상 파일 |
| `line` | `.line` 또는 `.original_line` | 수정 대상 라인 |
| `body` | `.body` | 리뷰 내용 |
| `in_reply_to_id` | `.in_reply_to_id` | null이면 최상위 코멘트, 값이 있으면 답글 |
| `user.login` | `.user.login` | 작성자 (자동 리뷰 vs 사람 구분용) |
| `created_at` | `.created_at` | 시간순 정렬용 |

#### 스레드 그룹핑

같은 `in_reply_to_id` 또는 같은 `id`를 공유하는 코멘트를 하나의 스레드로 묶는다.
스레드에 이미 `✅ 수정 완료` 또는 `수정완료`가 포함된 답글이 있으면 → 이미 처리된 것으로 간주하고 스킵.

#### 최상위 코멘트만 처리 대상

`in_reply_to_id`가 null인 코멘트만 리뷰 항목으로 취급한다. 답글은 맥락 파악용으로만 사용.

### Step 3: 리뷰 타당성 검토 및 작업 계획

각 리뷰 코멘트에 대해:

1. **해당 파일의 현재 코드를 읽는다** — `path`와 `line` 기준으로 실제 파일을 Read 도구로 확인.
2. **리뷰 내용과 현재 코드를 비교하여 타당성을 판단한다:**
   - 이미 수정된 경우 → `already_fixed` (스킵)
   - 코드가 존재하지 않거나 크게 달라진 경우 → `outdated` (스킵)
   - 리뷰 내용이 타당하고 수정 가능한 경우 → `will_fix`
   - 리뷰가 주관적 제안이거나 현재 컨텍스트에서 부적절한 경우 → `skip_reason`과 함께 스킵
   - 리뷰가 모호하거나 대규모 리팩터링을 요구하는 경우 → `needs_discussion` (스킵, 사유 기록)

3. **검토 결과를 사용자에게 테이블로 보여준다:**

```
| # | Severity | 파일 | 라인 | 에이전트 | 제목 | 판정 |
|---|----------|------|------|----------|------|------|
| 1 | 🔴 Critical | actions.ts | 132 | Senior FE | Open Redirect | will_fix |
| 2 | 🟠 Major | ProfileStep.tsx | 146 | Senior FE | canSubmit 허점 | will_fix |
| 3 | 🟡 Minor | IntroStep.tsx | 21 | SEO | h1 중복 | skip (noindex 적용 시 무관) |
```

4. **사용자의 확인을 기다린다.** 사용자가 특정 항목의 판정을 바꾸고 싶을 수 있다.

### Step 4: 코드 수정 실행

사용자가 승인한 작업 계획에 따라, `will_fix` 판정을 받은 항목을 severity 순서로 수정한다:

- **Critical → Major → Minor → Misc** 순서로 처리
- 각 수정 시 해당 파일을 Read한 뒤 Edit/Write 도구로 수정
- 수정 후 관련 파일의 타입 체크나 린트가 깨지지 않는지 간단히 확인
- 하나의 리뷰 항목 수정이 다른 항목에 영향을 줄 수 있으므로, 수정 후 후속 항목의 코드를 다시 읽어서 반영

#### 수정 시 주의사항

- 리뷰에서 제안한 정확한 수정안이 있으면 우선 따르되, 맹목적으로 복사하지 말 것. 현재 코드 맥락에 맞게 조정.
- 수정 범위는 리뷰가 지적한 부분으로 한정. 주변 코드를 불필요하게 리팩터링하지 않는다.
- import 추가/제거가 필요하면 함께 처리한다.

### Step 5: 수정 완료 댓글 게시

각 `will_fix` 항목의 수정이 완료되면, 해당 인라인 코멘트 스레드에 답글을 남긴다:

```bash
gh api repos/{owner}/{repo}/pulls/<PR_NUMBER>/comments --input - <<'JSONEOF'
{
  "body": "✅ 수정 완료\n\n{수정 내용 한 줄 요약}\n\n🤖 Generated by /roto-band-review-apply",
  "in_reply_to": <PARENT_COMMENT_ID>
}
JSONEOF
```

**중요**: `in_reply_to` 필드에는 해당 스레드의 **최상위 코멘트 ID**를 사용한다.

답글 본문 형식:

```
✅ 수정 완료

{수정 내용을 1-2줄로 요약}

🤖 Generated by /roto-band-review-apply
```

422 에러 시: 에러 내용을 로그하고 다음 항목으로 진행. 실패한 댓글은 요약 코멘트에 `❌ 댓글 실패` 표시.

### Step 6: PR 요약 코멘트 게시

모든 수정이 완료되면, PR에 요약 코멘트를 남긴다:

```bash
gh pr comment <PR_NUMBER> --body "$(cat <<'EOF'
## 🔧 코드 리뷰 반영 결과

### 반영된 항목
- ✅ [에이전트명] 이슈 제목 — `파일:라인` (수정 내용 요약)
- ✅ [에이전트명] 이슈 제목 — `파일:라인` (수정 내용 요약)

### 미반영 항목
- ⏭️ [에이전트명] 이슈 제목 — `파일:라인` (사유: 이미 수정됨 / 코드 변경됨 / 주관적 제안 / 대규모 리팩터링 필요)

### 변경 통계
- 수정: N건
- 스킵: N건
- 댓글 실패: N건

---
🤖 Generated by /roto-band-review-apply
EOF
)"
```

#### 에이전트명 추출 규칙

리뷰 코멘트 본문에서 에이전트명을 추출한다:
- `[🔴 Critical] [Senior FE]` → `Senior FE`
- `[🟠 Major] [UI]` → `UI`
- `[AI 코드리뷰]` → `AI 코드리뷰`
- 에이전트명이 없는 코멘트 (사람이 남긴 경우) → 코멘트 작성자의 GitHub username 사용

### Step 7: 대화 창 요약 출력

사용자에게 최종 결과를 간략히 알린다:
- 수정된 항목 수, 스킵된 항목 수
- 수정된 파일 목록
- 커밋 여부 확인 (사용자가 원하면 커밋)

## 엣지 케이스 처리

| 상황 | 처리 |
|------|------|
| 인라인 코멘트가 0건 | `리뷰 코멘트가 없습니다.` 출력 후 종료 |
| 모든 코멘트가 이미 처리됨 | `모든 리뷰가 이미 반영되었습니다.` 출력 후 종료 |
| 파일이 삭제된 경우 | `outdated` 처리 |
| PR이 이미 merge된 경우 | 경고 후 사용자 확인 |
| 현재 브랜치가 PR 브랜치와 다른 경우 | 사용자에게 알리고 브랜치 전환 여부 확인 |

## 도구 사용 규칙

Bash 대신 Claude 전용 도구를 우선 사용한다:

| 작업 | 사용할 도구 | Bash 사용 금지 |
|------|-----------|---------------|
| 파일 읽기 | **Read** | `cat`, `head`, `tail` |
| 파일 수정 | **Edit** | `sed`, `awk` |
| 파일 생성 | **Write** (디렉토리 자동 생성) | `mkdir -p` + `echo >` |
| 파일 검색 | **Glob** | `find`, `ls` |
| 코드 내용 검색 | **Grep** | `grep`, `rg` |

Bash는 아래 경우에만 사용한다:
- `gh` CLI 명령 (PR 조회, 코멘트 수집, 댓글 게시)

## 오류 처리

| 상황 | 메시지 |
|------|--------|
| GitHub 인증 실패 | `[ERROR] GitHub authentication required. Run: gh auth login` |
| PR 미존재 | `[ERROR] PR #<NUMBER>을 찾을 수 없습니다.` |
| 댓글 게시 실패 | 로그하고 다음 항목 진행, 요약에 실패 기록 |
| 코드 수정 중 에러 | 해당 항목 스킵, 사유 기록 |
